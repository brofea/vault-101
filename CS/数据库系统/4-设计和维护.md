# 数据库的设计和维护

数据库系统的设计和维护是数据库应用开发中的核心内容。良好的数据库设计可以减少数据冗余、提高查询效率、保证数据一致性；而完善的维护机制则可以确保数据库在运行过程中具备可靠性、安全性和可恢复性。

## 数据库设计

数据库设计是指根据用户需求，规划和建立数据库结构的过程。它的目标是设计出结构合理、数据一致、易于维护并能满足应用需求的数据库。

数据库设计通常包括以下几个阶段：

1. **需求分析**

   需求分析是数据库设计的起点，主要任务是了解用户需要存储哪些数据、数据之间有什么关系、系统需要支持哪些操作。

   例如，在学生管理系统中，需要分析学生、课程、教师、成绩等信息，以及它们之间的关系。

2. **概念结构设计**

   概念结构设计主要使用实体-联系模型，即 E-R 模型，对现实世界中的数据对象及其联系进行抽象。

   常见概念包括：

   - **实体**：现实世界中可以区分的对象，如学生、课程、教师。
   - **属性**：实体具有的特征，如学生的学号、姓名、年龄。
   - **联系**：实体之间的关系，如学生选修课程。

3. **逻辑结构设计**

   逻辑结构设计是将概念模型转换为某种数据库模型，关系数据库中通常转换为关系模式。

   例如：

   ```text
   学生（学号，姓名，性别，年龄，专业）
   课程（课程号，课程名，学分）
   选课（学号，课程号，成绩）
   ```

4. **物理结构设计**

   物理结构设计关注数据库在存储设备上的组织方式，包括索引设计、存储路径、数据分区等。

   合理的物理设计可以提高数据库访问速度。

5. **数据库实施与维护**

   数据库实施包括创建数据库、数据表、索引、视图等对象，并导入初始数据。

   数据库维护包括性能优化、数据备份、故障恢复、安全管理和完整性检查等。

## 事务处理

事务是数据库系统中一组逻辑操作单元。一个事务中的操作要么全部执行成功，要么全部不执行。

> 事务和程序的区别：事务是程序的一次执行，程序可能有多个事务

事务具有四个重要特性，通常称为 **ACID 特性**：

1. **原子性 Atomicity**

   事务中的所有操作是一个整体，要么全部完成，要么全部回滚。

   例如，银行转账时，从 A 账户扣款和向 B 账户加款必须同时成功。

2. **一致性 Consistency**

   事务执行前后，数据库都必须处于一致状态。

   例如，转账前后两个账户的总金额应该保持不变。

3. **隔离性 Isolation**

   多个事务并发执行时，一个事务的执行不应被其他事务干扰。

4. **持久性 Durability**

   事务一旦提交，其结果就应永久保存在数据库中，即使系统发生故障也不应丢失。

事务分为显式事务和隐式事务，具体来说，显式事务需要开发者明确定义；而隐式事务是单独的 SQL 语句自动作为一个事务执行

按照 Transact-SQL 标准，显示定义事务由 `BEGIN TRANSACTION` 开始，`COMMIT` 提交，`ROLLBACK` 回滚，示例如下：

```sql
BEGIN TRANSACTION;

UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;

COMMIT;
```

如果执行过程中出现错误，可以使用：

```sql
ROLLBACK;
```

事务处理的作用是保证数据库操作的可靠性和一致性。

## 数据库备份和恢复

数据库备份和恢复是数据库维护中的重要内容，用于防止数据丢失，并在系统故障后尽快恢复正常运行。

### 数据库备份

数据库备份是指将数据库中的数据复制到其他存储介质中，以便发生故障时恢复数据。

常见备份方式包括：

1. **完全备份**

   对整个数据库进行完整备份。

   优点是恢复简单；缺点是占用空间大、备份时间较长。

2. **增量备份**

   只备份自上次备份以来发生变化的数据。

   优点是节省空间和时间；缺点是恢复时需要依赖多个备份文件。

3. **差异备份**

   备份自上次完全备份以来发生变化的数据。

   它介于完全备份和增量备份之间，恢复复杂度适中。

4. **日志备份**

   备份数据库事务日志，可以用于恢复到某一具体时间点。

### 数据库恢复

数据库恢复是指当数据库出现故障时，根据备份文件和日志文件将数据库恢复到正确状态。

常见故障包括：

- 事务故障
- 系统故障
- 介质故障
- 计算机病毒
- 人为误操作

常见恢复方法包括：

- 利用备份文件恢复数据库
- 利用日志文件重做或撤销事务
- 恢复到某个时间点

数据库备份和恢复的目标是提高系统的可靠性和容灾能力。

#### 转储

转储（Dumping）是指将数据库中的数据复制到磁带或另一个磁盘上保存的过程，它是数据库恢复的基本技术之一。

根据转储时数据库是否处于运行状态，可以分为：

- **热备份 / 动态转储**：在数据库运行时进行备份，适用于对系统可用性要求较高的场景。
- **冷备份 / 静态转储**：在数据库关闭时进行备份，适用于维护窗口较长的系统。

根据转储的内容，可以分为：

- **海量转储**：备份整个数据库，包括数据、日志和配置文件。
- **增量转储**：只备份自上次转储以来发生变化的数据，适用于数据量较大的系统。

#### 日志

**日志**是动态转储的技术，它记录了数据库中发生的所有操作，包括数据修改、事务提交和回滚等

在修改数据时，永远先写日志再修改数据，这样即使系统发生故障，也可以通过日志来恢复数据。

当 UNDO 时，反向扫描日志，撤销未提交的事务；当 REDO 时，正向扫描日志，重做已提交的事务。

#### 恢复策略

- 事务故障恢复

- 系统故障恢复

- 介质故障恢复

## 并发控制

并发控制是指当多个用户或多个事务同时访问数据库时，数据库系统采取一定机制保证数据正确性和一致性。

如果没有并发控制，可能会出现以下问题：

1. **丢失修改**

   两个事务同时修改同一数据，后提交的事务覆盖了前一个事务的修改。

2. **脏读**

   一个事务读取了另一个事务尚未提交的数据。如果后者回滚，前者读到的数据就是错误的。

3. **不可重复读**

   一个事务多次读取同一数据，但由于其他事务修改并提交，导致前后读取结果不同。

4. **幻读**

   一个事务按照条件查询数据时，另一个事务插入或删除了符合条件的数据，导致前后查询结果数量不同。

常见并发控制方法如下

### 封锁和锁协议

通过加锁控制多个事务对数据的访问。

常见锁类型包括：

- **共享锁 S 锁**：用于读操作，多个事务可以同时加共享锁。
- **排他锁 X 锁**：用于写操作，只允许一个事务独占访问。

封锁协议由三级：

- **1级封锁协议**：事务修改数据前必须先加 X 锁
- **2级封锁协议**：基于 1 级协议，并且读取数据前必须先加 S 锁
- **3级封锁协议**：基于 2 级协议，並且在事务结束前不能释放锁

### 活锁和死锁

**活锁**是指两个或多个事务不断地相互让步，导致无法继续执行

> 比如，事务 A1、A2...A10 都需要访问同一资源，但 A1 释放锁后 A3 立刻加锁，A3 释放锁后 A4 立刻加锁，导致 A2 一直无法获得锁，形成活锁

**死锁**是指两个或多个事务相互等待对方释放锁，导致无法继续执行。

> 比如，事务 A 和 B 都需要持有 X 和 Y 锁才能开始工作，但事务 A 获得了 X 锁，事务 B 获得了 Y 锁，然后 A 等待 B 释放 Y 锁，B 等待 A 释放 X 锁，形成死锁。

### 两段锁协议

如果一个事务所有加锁都在释放锁的前面，则称之为两段事务，这种加锁的协议称为两段锁协议（Two-Phase Locking Protocol，2PL）。两段锁协议保证了事务的串行化。

- 扩展阶段 / 加锁阶段：只能申请锁，不能释放锁。
- 紧缩阶段 / 解锁阶段：只能释放锁，不能再申请锁。

### 粒度和意向锁

**粒度**是指锁定数据的范围，可以是行级锁、页级锁、表级锁等。

在保证数据一致性的情况下，粒度越细，系统并发性能越高，但管理开销也越大，粒度越粗，管理开销越小，但并发性能也越低。

**意向锁**是 IBM DB2 数据库引入的一种锁类型，用于表示事务打算在某个粒度上加锁。它有三种类型：

- **意向共享锁 IS 锁**：表示它的下级粒度加上了共享锁。
- **意向排他锁 IX 锁**：表示它的下级粒度加上了排他锁。
- **共享意向排他锁 SIX 锁**：表示它的下级粒度加上了共享锁和排他锁。

然后规定一套锁兼容性规则，来判断不同类型的锁是否可以共存。这样既保证了数据的一致性，又提高了系统的并发性能。

### 其他

- **时间戳机制**

   根据事务开始时间为事务分配时间戳，按照时间顺序控制事务执行。
- **多版本并发控制 MVCC**

   为数据保存多个版本，使读操作和写操作尽量不互相阻塞。

并发控制的核心目标是提高系统并发性能，同时保证数据库一致性。

## 数据库安全性

数据库安全性是指保护数据库，防止非法访问、数据泄露、恶意修改和破坏。

数据库安全性主要包括以下内容：

1. **用户身份认证**

   通过用户名、密码、数字证书、生物识别等方式确认用户身份。

2. **权限控制**

   根据用户角色授予不同权限。

   常见权限包括：

   - 查询权限 `SELECT`
   - 插入权限 `INSERT`
   - 修改权限 `UPDATE`
   - 删除权限 `DELETE`
   - 管理权限 `CREATE`、`DROP`、`GRANT`

   示例：

   ```sql
   GRANT SELECT, INSERT ON student TO user1;
   REVOKE INSERT ON student FROM user1;
   ```

3. **角色管理**

   将权限分配给角色，再将角色分配给用户，便于统一管理权限。

4. **数据加密**

   对敏感数据进行加密存储或传输，防止数据泄露。

5. **审计机制**

   记录用户对数据库的访问和操作，便于追踪异常行为。

6. **视图机制**

   通过视图限制用户只能访问部分数据。

   例如，只允许普通用户查看学生姓名和专业，而不能查看身份证号等敏感信息。

数据库安全性的目标是保证数据的保密性、完整性和可用性。

## 数据库完整性

数据库完整性是指数据库中数据的正确性、有效性和一致性。

完整性约束可以防止错误数据进入数据库。

常见完整性约束包括：

1. **实体完整性**

   实体完整性要求每个表必须有主键，且主键不能为空、不能重复。

   示例：

   ```sql
   CREATE TABLE student (
       student_id INT PRIMARY KEY,
       name VARCHAR(50)
   );
   ```

2. **参照完整性**

   参照完整性用于保证表与表之间关系的正确性，通常通过外键实现。

   示例：

   ```sql
   CREATE TABLE score (
       student_id INT,
       course_id INT,
       grade INT,
       FOREIGN KEY (student_id) REFERENCES student(student_id)
   );
   ```

3. **用户定义完整性**

   用户定义完整性是根据具体业务规则设置的约束。

   例如：

   - 年龄必须大于 0
   - 成绩必须在 0 到 100 之间
   - 邮箱格式必须合法

   示例：

   ```sql
   CREATE TABLE score (
       grade INT CHECK (grade >= 0 AND grade <= 100)
   );
   ```

4. **非空约束**

   要求某些字段不能为空。

   ```sql
   name VARCHAR(50) NOT NULL
   ```

5. **唯一约束**

   要求某些字段的值不能重复。

   ```sql
   email VARCHAR(100) UNIQUE
   ```

6. **默认值约束**

   当用户没有输入某字段值时，系统自动使用默认值。

   ```sql
   status VARCHAR(20) DEFAULT 'active'
   ```

数据库完整性的作用是从数据库层面保证数据质量，减少错误数据，提高系统可靠性。

## 小结

数据库的设计和维护贯穿数据库系统的整个生命周期。数据库设计决定了系统的数据结构和运行基础；事务处理、并发控制、备份恢复、安全性和完整性则保证数据库在实际运行中的正确性、可靠性和安全性。
